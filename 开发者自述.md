# 开发者自述

## 为什么我要写这个工具

**问题背景**

2025年4月，我被派驻坦桑尼亚北新建材海外基地，担任工艺技术岗位。坦桑基地面临的现实是：

- **技术依赖瓶颈**：生产线上的故障诊断高度依赖中方技术人员的个人经验。一旦人员轮换回国，知识随之流失，新来者需要重新踩坑。
- **知识分布碎片化**：94项产线故障的根因、应对措施分散在12个月的月度总结中，查找效率低下。本地员工遇到问题时既看不懂中文资料，也无法快速定位解决方案。
- **海外条件限制**：坦桑电网不稳定、天然石膏品位波动大、雨季潮湿——这些问题在国内基地很少出现，意味着"国内经验"不能直接复制，需要一套面向当地特殊工况的知识体系。

**核心认知：AI 的价值不在于替代人，而在于让知识不再依赖特定的人**

在坦桑工作的这一年，我深刻体会到：工厂最脆弱的地方不是设备会坏，而是"知道怎么修的人走了以后怎么办"。

这个工具试图解决的核心问题是——**知识的组织化沉淀**。

过去，94项故障被记录为12份月度报告的附录，只有作者本人能快速检索。现在，它们被结构化为机器可读的故障知识库，任何技术人员输入症状关键词，系统就能匹配出最可能的根因和应对措施。

这不是一个"智能诊断系统"——它不会替代技术人员的判断。它是一个"经验放大器"：让新手能迅速站上前人积累的肩膀，让老师傅的经验不再随他一起退休，让本地员工在凌晨三点闪电断电时，不再因为联系不上中方技术人员而束手无策。

**技术理念**

1. **结构化即智能**：将非结构化的月度总结文本转化为结构化的故障-症状-根因-措施四元组，这是比任何算法优化都更重要的第一步。
2. **关键词匹配 + 权重排序**：针对石膏板产线的特定术语（如"倒边漏浆""过火""AIII相""板软"）建立领域词典，用加权匹配替代模糊搜索。
3. **渐进式知识更新**：每解决一个新故障，只需在 JSON 知识库中追加一条记录，工具自动纳入诊断范围。
4. **双语可扩展**：故障知识库设计为中英文键值对，未来可快速翻译为斯瓦希里语，实现真正的属地化知识传递。

**从经验管理到知识治理**

这个工具不只是一个 Python 脚本。它代表一种工作方式的转变：

| 传统模式 | 组织化模式 |
|---------|----------|
| 故障经验记在师傅脑子里 | 故障经验登记在知识库中 |
| 新人靠"跟班"学习 | 新人靠"检索+验证"学习 |
| 知识随人员流动而流失 | 知识随使用频次而增值 |
| 本地员工只能做执行者 | 本地员工可以做初级诊断者 |

**写给未来**

我希望有一天，坦桑基地的本地技术员打开这个工具，输入"板边黄"，系统告诉他：先查干燥机空板段、再查过火程度、最后按标准流程调整控温——就像一年前的中方师傅在身边一样。

这不是 AI 取代了师傅。这是 AI 让师傅的经验不再被海风带走。
